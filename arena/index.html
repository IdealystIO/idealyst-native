Arena

A head-to-head benchmark of UI frameworks rendering the same screen, doing the same work, measured by the same instrumentation. Read the spec for what we measure and why.

Test: 1000-row list, theme toggle, 250ms transition. Reported per toggle: apply, first paint, frames, avg fps, worst frame.

Baselines

  • vanilla · CSS variables
    Theoretical ceiling. Toggle = flip a className on <html>. No JS work scales with row count.
  • vanilla · per-element classes
    Mirrors the framework's web backend: mint fresh CSS rules and setAttribute 2000+ times per toggle.

React

  • react · naive (inline styles, re-render)
    useState(theme) at the top, threaded down via props. Every row re-renders on toggle.
  • react · idiomatic (CSS variables, no re-render)
    Same theme model as the vanilla ceiling. React commits a single className change; rows are inert.

idealyst-native

  • idealyst-native · framework + web backend
    Rust → WASM. set_theme(...) fires the framework's per-node style effects, which point classNames at pre-generated CSS rules. Requires a one-time wasm-pack build — see the variant's README.

How to read this

Open two tabs and toggle each a few times. The numbers in #stats reset on every click. The apply column is the synchronous JS cost — that's what the framework is paying. The worst frame column is what the user actually feels.
