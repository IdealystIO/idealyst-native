# Refs

A **Ref** is how you get a programmatic handle to a mounted thing
— a button, a view, a custom component — so you can call methods
on it imperatively. You'll reach for one whenever you need to
**trigger an action** on the thing rather than describe it through
reactive state.

The most common case isn't a built-in primitive. It's giving a
component you wrote an imperative surface — "reset this form,"
"scroll this list to the top," "focus this input," "play this
animation" — and calling it from a parent. This page leads with
that pattern because it's the one app code actually uses every
day.

## The two flavors

Refs come in two shapes:

1. **Component refs** — a handle to one of your own
   `#[component]` functions, generated by its `methods!` block.
   Used to call imperative methods you declared on the component.
2. **Primitive refs** — a handle to a built-in primitive
   (`ButtonHandle`, `ViewHandle`, etc.). Used to trigger a click,
   read a frame, scroll a list to the top.

The mechanics are the same in both cases. A `Ref<H>` is a slot,
`.bind(ref)` attaches it to the mounted thing, `.with(|h| ...)`
calls into the handle. The two flavors differ only in **what**
the handle exposes.

## Component refs

This is the pattern you'll reach for most. A parent component
wants to imperatively trigger something inside a child — reset a
form, scroll a list, focus an input, play an animation. Instead
of routing the action through a prop and a signal flip, the
child **exposes the action as a method** and the parent **calls
it directly**.

### Declaring methods on a component

Inside a `#[component]` function body, a `methods!` block
declares the imperative methods the component exposes:

```rust
use framework_core::{component, signal, ui, Bindable, Primitive};

pub struct CounterProps {
    pub initial: i32,
}

#[component]
pub fn counter(props: &CounterProps) -> Bindable<CounterHandle> {
    let value = signal!(props.initial);

    methods! {
        fn reset(&self) {
            value.set(0);
        }
        fn bump_by(&self, n: i32) {
            value.update(|v| *v += n);
        }
        fn set_to(&self, n: i32) {
            value.set(n);
        }
    }

    ui! {
        View {
            Text { format!("Count: {}", value.get()) }
        }
    }
}
```

Three things change versus a regular component:

1. **The `methods!` block declares the imperative surface.** Each
   method takes `&self` and zero or more typed parameters, and
   runs against captures from the component's scope — here, the
   `value` signal.
2. **The return type is `Bindable<CounterHandle>`** instead of
   `Primitive`. The macro derives the handle name from the
   function name (`counter` → `CounterHandle`), so you don't
   write the struct yourself.
3. **The framework generates `CounterHandle`** with one method
   per declared `fn`. Calling `handle.bump_by(5)` invokes the
   closure body that captured `value`, runs in the component's
   scope, and updates the signal. Cascades fire normally.

Method bodies have access to anything captured from the
surrounding scope — signals, closures, other refs. Writes
propagate through the reactive substrate exactly the way they
would if the same write happened inside an `on_click`.

### Capturing the handle in a parent

The parent allocates a `Ref<CounterHandle>` and binds it to the
child:

```rust
use framework_core::Ref;

#[component]
pub fn parent_app() -> Primitive {
    let counter_ref: Ref<CounterHandle> = Ref::new();

    ui! {
        counter(initial = 10).bind(counter_ref)

        Button(
            label = "Reset",
            on_click = move || { counter_ref.with(|h| h.reset()); },
        )

        Button(
            label = "Add 5",
            on_click = move || { counter_ref.with(|h| h.bump_by(5)); },
        )
    }
}
```

What happens at runtime:

1. `Ref::new()` allocates a slot in the parent's scope. The slot
   is empty.
2. The framework mounts the counter. When the child's body has
   finished running and the handle has been constructed, the
   framework fills `counter_ref`'s slot.
3. **Before mount**, `counter_ref.with` returns `None` silently.
   **After mount**, it runs the closure with the handle.
4. Calling `handle.bump_by(5)` invokes the captured closure,
   which updates `value`. The reactive text inside the counter
   re-runs and updates on screen.

### Sibling-to-sibling through a parent

Two children of the same parent can talk to each other through
a ref the parent holds:

```rust
let target: Ref<CounterHandle> = Ref::new();

ui! {
    counter(initial = 0).bind(target)

    Button(
        label = "Set target to 100",
        on_click = move || { target.with(|h| h.set_to(100)); },
    )
}
```

The button doesn't know about the counter directly — it knows
about the ref. The ref is the wire that threads imperative
reach across the tree.

### Method vs signal prop — when to pick which

A natural question: "Should `reset` be a method, or should I
take a `Signal<bool>` prop the parent flips to trigger reset?"

**The method wins when:**
- The action is genuinely fire-and-forget — "do this thing now."
- Multiple call sites need to invoke it from different points in
  the tree.
- The action involves more than just changing one piece of state
  — focus shifts, animations, scroll position changes.

**The signal wins when:**
- The trigger is itself a piece of state worth observing.
- Sibling components also care about the same state and should
  react to it.

In practice, most apps mix both: signals for state that drives
UI, methods for actions the parent dispatches.

## Primitive refs

The framework's built-in primitives expose handles the same way.
Same shape — allocate a `Ref<H>`, bind, call methods:

```rust
let btn: Ref<ButtonHandle> = Ref::new();

ui! {
    Button(label = "Original", on_click = on_click).bind(btn)

    Button(
        label = "Trigger the original",
        on_click = move || { btn.with(|h| h.click()); },
    )
}
```

The handle types you'll meet, with the most-used method per
handle:

- **`ButtonHandle`** — `.click()` fires the press
- **`PressableHandle`** — `.click()` fires the press
- **`ViewHandle`** — `.frame()`, `.absolute_frame()` for geometry
- **`LinkHandle`** — `.click()` for trigger-on-keystroke patterns
- **`NavigatorHandle`** / **`TabsHandle`** / **`DrawerHandle`** —
  navigation commands (covered in [Navigation](#))
- **`IconHandle`** — start a stroke animation imperatively
- **`OverlayHandle`** / **`AnchoredOverlayHandle`** /
  **`PresenceHandle`** — overlay control

The rest (`TextHandle`, `ImageHandle`, `TextInputHandle`,
`ToggleHandle`, `SliderHandle`, `WebViewHandle`, `VideoHandle`,
`ActivityIndicatorHandle`, `VirtualizerHandle`,
`GraphicsHandle`) exist but their method surfaces are still being
filled in. New operations land on those traits without app code
needing to change.

### Reading geometry

The most common primitive-ref use is reading a view's bounding
rect:

```rust
let row: Ref<ViewHandle> = Ref::new();

ui! {
    View { /* content */ }.bind(row)

    Button(label = "Where am I?", on_click = move || {
        if let Some(rect) = row.with(|h| h.frame()).flatten() {
            println!("row is at {:?}", rect);
        }
    })
}
```

Two methods:

- **`.frame()`** — rect in the *parent's* coordinate system
- **`.absolute_frame()`** — rect in viewport (window) coordinates

Both return `Option<ViewportRect>` because layout might not have
happened yet, or the backend might not expose geometry. Backends
that do: web (`getBoundingClientRect`), iOS (`UIView.frame` +
`UIView.convert`), Android (`View.getLocationOnScreen`).

### Anchoring overlays

`ButtonHandle`, `PressableHandle`, and `ViewHandle` all implement
the `AnchorableHandle` trait. `AnchoredOverlay` reads it to
position itself relative to the anchor:

```rust
let trigger: Ref<ButtonHandle> = Ref::new();
let is_open = signal!(false);

ui! {
    Button(
        label = "Open menu",
        on_click = move || is_open.update(|v| *v = !*v),
    )
    .bind(trigger)

    if is_open.get() {
        AnchoredOverlay(
            target = trigger,
            side = ElementSide::Below,
            align = ElementAlign::Start,
        ) {
            // menu content
        }
    }
}
```

The overlay tracks the anchor through scrolls, layout shifts,
and orientation changes — same rect machinery as
`.absolute_frame()`, read continuously.

## The Ref<H> API

```rust
let r: Ref<H> = Ref::new();

r.with(|h| h.some_method());      // Option<R> — runs the closure if mounted
r.is_mounted();                    // bool
r.get();                           // Option<H> — clone of the handle if mounted
```

A `Ref<H>` is a `Copy` token, same shape as `Signal<T>`. Pass it
into closures, into child components, anywhere, without
`.clone()`. The token indexes into the reactive arena; the
storage is shared.

Allocate refs early — typically right at the top of a component
— and bind them inline in the `ui!` block. The slot stays empty
until the primitive (or component) mounts.

## Binding via `.bind`

Every primitive's builder and every `Bindable<H>`-returning
component exposes `.bind(ref)`:

```rust
View { ... }.bind(view_ref)
Button(label = "...", on_click = ...).bind(button_ref)
ScrollView { ... }.bind(scroll_ref)
counter(initial = 0).bind(counter_ref)
Navigator::new(&HOME).screen(...).bind(nav_ref)
```

The handle type has to match the primitive (or component) kind
— `.bind` is type-checked. Passing a `Ref<ButtonHandle>` to a
View's `.bind` is a compile error.

## Lifecycle

A ref's slot is empty until the primitive mounts. Between
`Ref::new()` and the first mount, `.with` returns `None` and
`.is_mounted()` is `false`. Once mounted, the slot stays filled
until the primitive unmounts (its scope drops). After unmount,
the slot empties.

Reactive subtrees produce common lifecycle cases:

- A ref bound to a primitive inside a `when()` — the slot empties
  when the condition flips false, fills again when it flips back.
- A ref allocated inside a screen's scope — when the screen
  pops, the slot is freed along with everything else in that
  scope.

You don't have to handle these explicitly. `.with` returns `None`
rather than panicking when the slot is empty, and the lifecycle
is owned by the scope, not by you.

## Refs vs Signals

Two distinctions worth keeping straight:

- **Signal `<T>`** — reactive state. Reads inside tracked
  contexts subscribe; writes notify subscribers. Use for
  anything that changes over time and drives UI.
- **Ref `<H>`** — an imperative handle slot. Filled at mount,
  emptied at unmount. Reading it doesn't subscribe to anything.
  Use to invoke methods on a mounted thing.

Both live in the reactive arena and share the Copy-token shape.
They solve different problems and coexist freely — you'll often
fire a ref-driven action from inside a signal-driven event
handler.
